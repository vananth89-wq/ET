## 2. Architecture & Data Model

### Key Tables (relevant to this session)

**`employee_bank_accounts`** (mig 273)
- `id` uuid PK, `bank_account_group_id` uuid (groups history rows), `employee_id` uuid FK employees

**`employee_bank_attachments`** (mig 273)

- `relationship_type` text (ref_id from `DEPENDENT_RELATIONSHIP_TYPE` picklist), `dependent_name` text, `date_of_birth` date, `gender` text ('Male' or 'Female')
- `insurance_eligible` boolean DEFAULT false, `effective_from` date, `effective_to` date DEFAULT `'9999-12-31'`, `is_active` boolean DEFAULT true
- `inactive_at` timestamptz, `inactive_by` uuid, `created_by` uuid, `updated_by` uuid, `created_at` timestamptz, `updated_at` timestamptz
- **Active record:** `is_active = true AND effective_to = '9999-12-31'`
- **Historical amendment:** `is_active = true AND effective_to < '9999-12-31'`
- **Explicitly removed:** `is_active = false`

**`employee_dependent_attachments`** (mig 289)
- `id` uuid PK, `dependent_code` text, `employee_id` uuid FK, `dependent_id` uuid FK
- `document_type` text (ref_id from `DEPENDENT_DOCUMENT_TYPE` picklist — **mandatory** as of this session)
- `file_name` text, `original_file_name` text, `file_path` text, `mime_type` text, `file_size` bigint

- `uploaded_at` timestamptz, `created_at` timestamptz, `updated_at` timestamptz

### Effective-Dating Pattern

Both bank accounts and dependents use the same pattern:
- One "open-ended" row per logical entity: `effective_to = '9999-12-31' AND is_active = true`
- Amendment creates new row, closes old row by setting `effective_to = new_effective_from - 1 day`
- Removal sets `is_active = false` (either in-place for path 1, or via terminal row for path 2)
- `isActive` check (strict AND): `is_active === true && effective_to === '9999-12-31'`

### Key RPCs (written/modified this session)

**`upsert_bank_account(...)`** → jsonb
- SECURITY DEFINER. Dual-path: PATH A (direct write if no workflow assigned), PATH B (stages in `workflow_pending_changes`).
- Date rules (standard ESS, not new hire, not exempt role): submit on/before 15th; `effective_from` must be exactly 1st of current month.
- Exempt roles: `bank_exceptions`, `admin`, `hr` bypass all date rules.
- New hires bypass all date rules.
- Country mandatory fields: IND→IFSC, LKA→branch_code, PAK/SAU→IBAN.

**`get_employee_bank_accounts(p_employee_id uuid, p_include_history boolean)`** → jsonb[]
- Returns bank accounts with attachments. History included when `p_include_history = true`.

**`upsert_dependent(...)`** → jsonb
- SECURITY DEFINER. Dual-path: direct write or workflow staging.
- `p_dependent_code = NULL` → new dependent; non-null → amendment.
- Workflow detection: uses `resolve_workflow_for_submission('profile_dependents', auth.uid())` — NOT a direct `workflow_templates` query (that was the mig 290 bug, fixed in mig 300).

**`remove_dependent(p_employee_id uuid, p_dependent_code text, p_removal_date date)`** → jsonb
- SECURITY DEFINER. Dual-path. PATH A path 1: if `effective_from >= removal_date`, sets `is_active = false` in-place. PATH A path 2: closes current row, inserts terminal row with `is_active = false, effective_from = removal_date`.

**`apply_profile_pending_change()`** — TRIGGER FUNCTION on `workflow_pending_changes`
- Fires AFTER UPDATE when status → `'approved'`.
- Handles: `profile_personal`, `profile_contact`, `profile_address`, `profile_passport`, `profile_emergency_contact`, `profile_bank`, `profile_dependents`.
- `profile_bank` branch: blocks approval after day 20 unless `has_role('bank_exceptions')` OR `has_role('admin')` OR `has_role('hr')` (added mig 299).
- `profile_dependents` branch: handles add/amend/remove operations.

**`get_profile_workflow_gates()`** → jsonb
- Returns `{ gated_modules: text[], pending_counts: jsonb, instance_ids: jsonb, is_bank_exception: boolean }`.
- `is_bank_exception = has_role('bank_exceptions') OR has_role('admin') OR has_role('hr')`.
- Covers all 9 profile modules including `profile_bank` and `profile_dependents` (added mig 291).

**`get_employee_pending_sections(p_employee_id uuid)`** → text[]
- SECURITY DEFINER. Returns array of `module_code` strings with pending changes for a given employee.
- Requires `hire_employee.view` permission. Used by `EmployeeEditPanel`.
- Maps to section IDs via `MODULE_TO_SECTION` constant in frontend.

## 3. Business Rules & Requirements

### Bank Account Effective-Dating Rules

| Rule | Detail |
|---|---|
| Submission window | On/before the 15th of the month |
| `effective_from` constraint | Must be exactly the 1st of the current month (not past, not future) |
| Approver window | Must approve by the 20th of the month |
| Exempt roles | `bank_exceptions`, `admin`, `hr` bypass both cutoffs |
| New hire exemption | All date rules skipped for new hires |

### Dependent Module Rules

- `document_type` on attachments is mandatory (validation added this session)
- `effective_from` is the first field in the form
- `isActive` uses strict AND: `is_active === true && effective_to === '9999-12-31'`
- Historical amendment rows (closed by amend) are NOT shown as "removed" — only `is_active = false` rows are "removed"

### Workflow Assignment

- `profile_dependents` module uses the "Hire" workflow template (assigned via Manage Assignments UI)
- The workflow lookup uses `resolve_workflow_for_submission()` which reads `workflow_assignments` table — NOT `workflow_templates.module_code`

Module permission mapping in `ApproverInbox.tsx`:
- `profile_dependents` → `dependents.edit`
- `profile_bank` → `bank_accounts.edit`

### Module Display Labels (`MODULE_LABELS` in `ApproverInbox.tsx`)

```ts
profile_bank:       'Bank Account Update'
profile_dependents: 'Dependents Update'
profile_personal:   'Profile – Personal Info'
// etc.
```

## 6. Recent Decisions

1. **`resolve_workflow_for_submission` vs direct template query** — mig 290 queried `workflow_templates WHERE module_code = 'profile_dependents'` which always returned NULL because the "Hire" template has `module_code = 'employee_hire'`. Fixed in mig 300 to use `resolve_workflow_for_submission()` which reads `workflow_assignments`. This is the same pattern as bank accounts.

2. **Inline amend in DependentCard** — User wanted amend form to open inside the card (not below all cards). Implemented by moving `amendOpen` state inside DependentCard, toggling between field view and `AddDependentForm` as card body.

3. **History toggles only the panel, no "back" button** — Clock icon is the sole toggle. When clicked, field view hides and history panel shows; click again to return. No separate "Back to record" button.

4. **`isActive` strict AND** — Both bank and dependents use `is_active === true && effective_to === '9999-12-31'`. This ensures amendment-closed rows (is_active=true, effective_to=past) are NOT shown as "removed".

5. **Document type mandatory** — Every attachment must have a document type before saving. Red border + inline error on the selector; validation in `handleSave` blocks save if any attachment is missing type.

6. **DependentsEnrichment as separate component** — Couldn't use `useState` inside `if (moduleCode === 'profile_dependents')` block in `ProfileEnrichment` due to React hooks rules. Extracted into its own top-level component so it can have proper state for attachment management.

7. **Attachment edit in workflow inbox** — When initiator is in "Sent Back" edit mode (or approver uses Update), attachments in `DependentsEnrichment` show remove X button and document type selector. New files can be uploaded via `hr-attachments` bucket. Changes sync to `editValues['attachments']` as JSON string.

8. **`flex: 1` removed from `.wfi-respond-btn`** — "Respond & Resume" button was spanning full width. Removed `flex: 1`, added `white-space: nowrap` so it sits at natural content width like the Approve/Reject buttons.

9. **ProwessLogo bucket secured** — Added "Admin upload only" (INSERT, requires `is_super_admin()`) and "Public logo read" (SELECT) policies. Was at 0 policies with PUBLIC access — anyone could upload any file type.

10. **`profile_bank` and `profile_dependents` missing from `MODULE_EDIT_PERMISSION`** — `can('profile_dependents.edit')` returned false (no such permission) so the Update button never showed. Fixed by mapping to `dependents.edit` and `bank_accounts.edit`.

### `fmtDepValue` in `ApproverInbox.tsx`

```typescript
function fmtDepValue(key: string, val: unknown): string {
  if (val == null || val === '') return '—';
  if (typeof val === 'boolean' || val === 'true' || val === 'false')
    return (val === true || val === 'true') ? 'Yes' : 'No';
  const s = String(val);
  if (DEP_REL_LABELS[s]) return DEP_REL_LABELS[s];
  if (/^\d{4}-\d{2}-\d{2}/.test(s))
    return new Date(s + 'T00:00:00').toLocaleDateString('en-GB', { day: '2-digit', month: 'short', year: 'numeric' });
  return s || '—';
}
```
