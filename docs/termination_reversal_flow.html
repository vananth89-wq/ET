Termination Reversal — How it works

Full flow from approval through employment slice undo

Full Flow
Employment Slices
Gaps to fix
1
Termination is APPROVED + jobs fire
apply-termination-approval Edge Function runs (on approval or via daily cron when LWD ≤ today):
  • Close open employment slice → effective_to = last_working_date
  • Insert new Inactive slice → effective_from = last_working_date + 1
  • Set employees.status = 'Inactive' → triggers sync → auto-closes JR sets
  • Stamp scheduled_executed = true
2
HR submits a Reversal
Only allowed when termination workflow_status = 'APPROVED'. HR provides reversal_reason + comments (min 20 chars) + optional attachments.
  • Inserts row into employee_termination_reversals (DRAFT → PENDING)
  • Submits to termination_reversal workflow template for approval
  • Unique constraint prevents two active reversals on same termination
3
Reversal goes through workflow approval
Standard workflow: approver reviews reason + comments. On approve:
  • Reversal workflow_status → APPROVED
  • Original termination workflow_status → REVERSED (unlocks the unique partial index — allows a fresh termination in future)
4
apply-termination-reversal Edge Function fires
Undoes the employment changes made by step 1:
  • Delete Inactive slice (effective_from = last_working_date + 1, status = Inactive)
  • Reopen prior slice → effective_to = '9999-12-31', is_active = true
  • Set employees.status = 'Active'
  • JR sets are NOT auto-restored — HR must reassign manually (locked design decision)
5
Employee is Active again
Employment history is intact. The REVERSED termination record and the APPROVED reversal record both remain as an audit trail. A new termination can be submitted for the same employee if needed.

Edge Function apply-termination-reversal uses term.termination_date — will break after rename
The reversal function uses term.termination_date to locate the employment slices to undo: deletes slice where effective_from = termination_date + 1 and reopens slice where effective_to = termination_date. After renaming to separation_date / last_working_date, both lookups will fail silently — the employee will remain Inactive with no error thrown. Must update both references to last_working_date.
Edge Function apply-termination-approval also uses termination_date — same break
The approval function closes the employment slice at termination_date and inserts the Inactive slice at termination_date + 1. After rename, this must use last_working_date. This is consistent with the design intent — jobs should key on last_working_date as the real operational anchor.
Design Reversal after scheduled_executed=true — works but JR sets are lost
The reversal correctly undoes employment slices even after scheduled_executed = true because it operates on the slice data directly. However, JR sets that were auto-closed at termination time are NOT restored. This is a documented locked decision (§1 #2), but HR must know to manually reassign the employee to their manager hierarchy after a reversal. Consider adding a checklist item or alert in the notification.
Design Reversal allows fresh termination — correctly handled
Setting the original termination to REVERSED releases the unique partial index WHERE workflow_status IN ('PENDING','APPROVED'). This correctly allows HR to submit a new termination for the same employee after a reversal, without any schema changes needed.
