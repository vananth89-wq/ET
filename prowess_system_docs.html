⚠ RLS Design Note — Circular Dependency (mig 174)
wf_instances_select needs three access paths: submitter, wf_manage.view, and assigned approver. The approver clause originally used an inline EXISTS (SELECT 1 FROM workflow_tasks WHERE assigned_to = auth.uid()). This created a circular dependency: wf_tasks_select queries workflow_instances (to let submitters see their tasks), and wf_instances_select queries workflow_tasks (to let approvers see instances) — each policy triggering the other infinitely.

Fix (mig 174): The inline EXISTS was replaced with is_wf_task_assignee(instance_id) — a SECURITY DEFINER function that reads workflow_tasks as the function owner, bypassing RLS and breaking the loop. The function still uses auth.uid() internally so no permissions are widened.

The approver clause must stay: vw_wf_pending_tasks (the approver inbox view) is a plain CREATE VIEW that JOINs workflow_instances. If approvers cannot SELECT that table, the JOIN returns nothing and the inbox goes blank. Removing the clause was investigated and ruled out — see mig 174 comment.

📖 Policy Function Glossary

Every unique pattern you'll see in the POLICY cells above — explained simply, with a real-world analogy.

user_can('module', 'action', NULL)
Path B — Module-level permission check
Asks: "Does this user hold the module.action permission at all?" No scope involved — it's a yes/no based purely on what's in their permission set. The NULL means "don't filter by target group."
🏢 Analogy: Your office keycard is programmed for "Finance Room access." The door doesn't check which desk you're going to — it just checks if Finance Room is on your card. If yes, door opens for every desk inside.
user_can('module', 'action', employee_id)
Path D — Scoped permission check
Asks: "Does this user have module.action AND does their target group include this specific employee?" Both must be true. Same permission code, but evaluated per-row against the requesting user's allowed scope (self / team / department / country).
🏢 Analogy: Your manager badge says "can view expense reports." But at the door of each report, the database checks: "Is this employee actually in your team?" Your badge alone isn't enough — the report also has to belong to someone you manage.
user_can('module', 'lookup', NULL)
Lookup — Read-only dropdown access
A lighter version of view. Grants SELECT on reference tables (currencies, picklists, projects, departments) just enough to populate dropdowns and autocomplete fields. You can't manage the data — you can only read the list.
🏢 Analogy: You're filling out a form and need to pick a department from a dropdown. You're not the Departments Admin — you can't add or edit departments — but the system still lets you read the list so your form works.
is_super_admin()
Super admin hard bypass
Checks if the calling user's ID is in the super_admins table. If yes, the RLS policy passes immediately — no permission matrix check needed. Only a handful of tables use this as a clause (system config, super_admins itself). Not the same as having a powerful role.
🏢 Analogy: The building master key. Works on every lock regardless of which department you belong to. Very few people have it, and it's managed separately from the normal keycard system.
auth.uid() = id  /  user_id = auth.uid()
Self-access — own row only
Compares the logged-in user's Supabase auth ID against a column on the row. If they match, access is granted — no permission needed. Used on tables like profiles, audit_log, workflow_tasks so users can always see their own data.
🏢 Analogy: Your payslip has your name on it. You can see it because it's yours — regardless of what permissions your role has. No badge needed; it just checks if your name matches.
auth.uid() IS NOT NULL
Any authenticated user
The loosest gate — if you're logged in at all, you pass. Used for tables where all employees need basic read access (e.g. user_roles SELECT so user_can() can look up your own permissions, target_groups to show non-system groups in the UI).
🏢 Analogy: The company cafeteria. If you have an employee badge (you're logged in), you can enter. There's no special "cafeteria permission" — showing up with a valid badge is enough.
service_role only
Backend internal writes — humans cannot trigger directly
Only the Supabase service role key (used by backend jobs, edge functions, pg_cron) can write to this table. Regular authenticated users have no INSERT/UPDATE/DELETE policy — any attempt is blocked at the DB level. The service role bypasses RLS entirely.
🏢 Analogy: The server room. Only IT systems staff with a special "service" badge can get in. Even the CEO can't walk in with their normal badge. Human employees never touch those servers directly.
Triggers / service_role
Written by DB triggers or backend jobs — never by a user action
The row is created automatically in response to something else happening (e.g. a trigger fires when you update a field, or a cron job runs nightly). There's no INSERT RLS policy for users because no user is supposed to ever write to this table directly.
🏢 Analogy: The building's automatic door-entry log. When you swipe your badge, the system writes a log entry automatically. You never type into that log yourself — it just happens behind the scenes.
Only via handle_new_user() trigger
Row created automatically on first sign-up
The handle_new_user() function is a Postgres trigger attached to Supabase's auth.users table. The moment a new user account is created, Postgres automatically calls it to insert a matching row in profiles. No human INSERT is ever needed.
🏢 Analogy: When HR adds a new employee to the HR system, the payroll system automatically creates their payslip record. Nobody in payroll has to manually create it — it's triggered by the hire event.
Managed by sync job (service_role)
Background job keeps this table in sync — not user-driven
A scheduled job (pg_cron or an edge function) periodically refreshes this table based on other data. Used for derived/denormalised tables that would be too expensive to compute on every query. Users read from it, but the backend owns all writes.
🏢 Analogy: The company org chart poster on the wall. It's printed and updated by IT every Monday. Employees read it, but nobody walks up and scribbles on it — the update happens on a schedule behind the scenes.
SECURITY DEFINER
Function runs as its owner — not as you
When a function is marked SECURITY DEFINER, it executes with the permissions of whoever created the function (typically a superuser), not the permissions of whoever is calling it. This lets the function read tables you'd normally never have access to — but only in a controlled, scoped way defined inside the function body.
🏢 Analogy: You ask the bank teller to check your balance. The teller uses their master terminal to look it up — you don't get access to the terminal yourself. The teller controls exactly what they show you. You asked; they did the lookup on your behalf.
SECURITY INVOKER
Function runs as you — your RLS still applies
The default for Postgres functions. When called, the function uses the calling user's own permissions. So if you try to JOIN two tables inside an SECURITY INVOKER view/function and you don't have SELECT on one of them, the join fails. Your RLS policies are still fully enforced inside the function.
🏢 Analogy: You log into the HR portal yourself and try to pull a colleague's record. The system checks your own permissions as you navigate. Nothing is elevated — if you don't have access, you get blocked just like anywhere else.
WITH CHECK (false)
Explicit block — no user can ever INSERT/UPDATE this table
WITH CHECK is the write-side of an RLS policy — it validates the row you're trying to write. Setting it to false means the check always fails, making it impossible for any authenticated user to write a row here directly. Used on audit tables and system logs where only triggers/service_role should ever write.
🏢 Analogy: The audit trail binder in the filing cabinet has a lock that no employee key fits. You can read the pages, but nobody can slip a new page in themselves. Only the official process can add pages.
Via parent expense report RLS
Child table inherits access from parent row
This table has no independent RLS check — instead, accessing it requires going through a JOIN to its parent table (e.g. expense_reports), and the parent's RLS policy does the gating. If you can see the parent report, you can see its child rows. If you can't, you can't — even if you know the child row's ID.
🏢 Analogy: A folder inside a locked filing cabinet drawer. The drawer itself has a lock (the parent's RLS). There's no separate lock on the folder inside. If you have the key to the drawer, you get the folder too.
Via parent report (draft only)
Child table inherits parent access — but only while the parent is a draft
Same as the parent-RLS pattern above, but with an additional status gate: the INSERT/UPDATE only works if the parent record is still in draft status. Once submitted/approved, the parent is locked and child rows can no longer be modified through normal user actions.
🏢 Analogy: You can add receipts to an expense claim while it's still on your desk (draft). Once you've handed it to your manager (submitted), the envelope is sealed — you can't slip more receipts in, even though you're the one who created it.
Via wf_submit() / wf_approve() / wf_reject() RPC only
Only a specific SECURITY DEFINER function is allowed to write here
The table itself allows any authenticated user to INSERT (open gate) — but in practice the only code that calls INSERT is the wf_submit() (or approve/reject) RPC function. The function is SECURITY DEFINER, so it runs under elevated permissions and can enforce business rules (valid workflow, correct state transitions) before writing.
🏢 Analogy: The "Submit for Approval" button on a form. Technically the submit endpoint accepts your request, but only the official Submit button wires it up correctly — it validates everything first. You can't POST directly to bypass the checks because the button handler does the real work.
get_record_history(p_entity_type, p_entity_id)
SECURITY DEFINER RPC — audit history without full audit log access
A controlled window into employee_audit_log. You call this function with a specific entity type and ID (e.g. an employee's UUID), and it returns only the change history for that record — nothing else. The function itself (not you) reads the audit log using elevated permissions, then filters to only what you're entitled to see based on your *.history permission.
🏢 Analogy: You ask the archivist: "Can I see the change history for employee #4521?" The archivist goes into the secure vault (which you can't enter), finds just that employee's file, and hands you only those pages. You never touch the vault.

Resolves the active workflow template steps for a given module code into a display-safe participant list for the WorkflowSubmitModal. The function reads workflow_steps, profiles, and employees using elevated permissions — bypassing the wf_templates.view RLS that normally gates those tables — and returns only the information needed to render the "Who will approve this?" preview.
Returns: jsonb array of { step_order, step_name, approver_type, approver_label } — no raw IDs exposed.

🏢 Analogy: Before submitting a form, you want to see the list of people who will review it. You can't read the approval rulebook yourself (wf_templates.view), but you can ask reception "Who needs to sign off on a leave request?" — reception checks the rulebook and reads you the names without handing you the book.

Part 7 — Design Observations

An objective review of the schema's architectural strengths, technical risks, and suggested improvements. Written for engineering leadership and future maintainers.

✅ What's Well Designed

Three-Layer Security Architecture
The system enforces authorization at three independent layers: DB RLS (user_can() on every table), UI gate (get_my_permissions() cached in React), and data scoping (get_target_population()). Compromise of any single layer doesn't bypass the others. This is enterprise-grade defense-in-depth.
Permission Set Abstraction
The PSA chain (role → permission_set → permissions) decouples role assignment from permission grants. Adding a new permission to a set immediately applies it to all roles holding that set — no per-role migration needed. This is exactly how enterprise RBAC systems (SAP, Workday) handle it.
Target Group Scoping
The target_groups + scope_type system elegantly handles "a manager can see only their direct reports" vs "finance can see everyone" within the same permission framework — without hardcoded role names in RLS policies. New scope types can be added without schema changes.
Immutable Audit Tables
RLS UPDATE/DELETE = false on audit tables is a strong compliance guarantee. SECURITY DEFINER trigger functions capture auth.uid() correctly even when called from service_role context. The dual-table approach (audit_log for permissions, employee_audit_log for PII changes) keeps concerns separated.
Employee Satellite Table Design
Splitting employees into 7 satellite tables (personal, contact, employment, addresses, emergency_contacts, identity_records, passports) enables per-module RLS at the field level. An employee can edit their own contact info without seeing another's identity documents. Rare in small-team HRIS systems.
Workflow Engine Decoupling
The workflow engine is generic (module_code + record_id) and not tied to expense reports. Any future module (leave requests, procurement) can reuse the same template/instance/task machinery. Template versioning (UNIQUE code+version) means in-flight approvals are not broken by template updates.

⚠️ Technical Risks

RLS Performance at Scale
user_can() is called on every row evaluated by RLS. For tables with 100k+ rows, this can result in thousands of function invocations per query. The function is STABLE and Supabase caches it per transaction, but complex queries (JOINs across 3+ tables) may still run user_can() many times. Benchmark at 50k+ employee rows before go-live.
target_group_members Sync Lag
The 'everyone' and 'custom' scope types rely on pre-computed target_group_members rows (synced by a pg_cron job). If the sync job fails or lags, newly onboarded employees won't appear in 'everyone' groups until the next run. This could silently lock out finance approvers from new employee data.
employee_status Title-Case Strings
'Draft', 'Incomplete', 'Active', 'Inactive' are title-case text values, not a PostgreSQL ENUM. This is inconsistent with expense_status (lowercase) and means a typo like 'active' will silently insert and bypass status checks. Should be migrated to a proper ENUM type.
No Row Count Guard on Satellite 1:1
employee_personal, employee_contact, employee_employment use employee_id as PK — correct. But employee_addresses, emergency_contacts, identity_records, passports use their own UUIDs, allowing multiple rows per employee. Application code must enforce "one address at a time" logic — there's no DB constraint preventing duplicates.
✓ Legacy role_permissions Removed (resolved in mig 146)
role_permissions table, sync triggers (trg_psi_sync, trg_psa_sync), and sync functions dropped. user_can() now reads permission_set_assignments exclusively. Verified by pre-flight DO block in mig 146.
ip_address in audit_log is App-Populated
audit_log.ip_address is a nullable text field populated by the application layer. This means IP logging depends on the API layer correctly passing the IP — it's not enforced at DB level. A misconfigured proxy could log incorrect IPs or leave the field NULL.
✓ workflow_pending_changes INSERT Hardened (resolved in mig 160 + 161)
What was the gap: Only workflow_pending_changes INSERT had an open gate — auth.uid() IS NOT NULL — meaning any authenticated user could directly INSERT a pending change row, bypassing wf_submit() business-rule enforcement. workflow_instances and workflow_tasks were already gated on admin permissions.

Mig 160: Changed wpc_insert to user_can('wf_manage','edit',NULL) — same pattern as workflow_tasks. wf_submit() is SECURITY DEFINER and bypasses RLS, so it is unaffected.

Mig 161: Wired profile_* modules into wf_sync_module_status(). Previously only expense_reports was handled — profile changes were approved in the workflow engine but workflow_pending_changes.status was never updated, so trg_apply_profile_pending_change never fired and the data never landed in satellite tables.

Overall Assessment

Prowess is built on a security-first foundation that is unusually sophisticated for a startup-stage HRIS. The three-layer enforcement model, audit immutability, and generic workflow engine would not look out of place in a Series B enterprise product. Legacy dead code (role_permissions table, sync triggers, hardcoded ESS seeds) has been removed — user_can() has a clean two-path structure backed solely by permission_set_assignments, and the Permission Matrix UI is the single source of truth for all permission configuration. Every expense_reports policy routes through user_can() Path D; access scope is fully controlled by target groups rather than hardcoded role checks.

Migrations 167–175 completed the profile-change workflow path end-to-end: ESS users can submit personal data changes for approval, approvers see their inbox correctly, and mid-flight editing is gated per step config. Two latent RLS circular dependency bugs were discovered and fixed (mig 064 pattern, now mig 174): workflow_tasks ↔ workflow_instances cross-table EXISTS subqueries each triggering the other's SELECT policy, resolved by SECURITY DEFINER helper functions that bypass caller RLS without widening permissions. The wf_sync_module_status() function now has a complete status mapping covering all engine states. Migration 175 closed a correctness gap in the client gate layer: useProfileWorkflowGates now calls get_profile_workflow_gates() — a SECURITY DEFINER RPC that uses resolve_workflow_for_submission() internally — so the UI gate and the submission layer always evaluate the same EMPLOYEE → ROLE → GLOBAL resolution for the current user. Gates are also refreshed on every edit-mode entry, eliminating stale-on-mount issues. The Approver Inbox renders a Salesforce-style before/after diff — same grid layout, old value struck through below the new value for any changed field. All changes are additive and backward compatible; rows submitted before mig 176 show no diff and behave identically to before. The codebase is in good shape for a controlled go-live with a pilot customer.

wf_notifications vs wf_notification_config
wf_notifications.view = notification queue monitor (read-only operational view). wf_notification_config.view/edit = template management screen. Template write RLS was moved from wf_notifications.edit → wf_notification_config.edit in mig 152.
employee_details appears twice

departments.view vs departments.lookup
departments.view gates the management screen (CRUD). departments.lookup gates read-only access to department dropdown in forms. A user can have lookup without view. Same pattern for picklists, projects, currencies.
wf_inbox vs wf_delegations
wf_inbox.view = approver permission; gates self-service delegation + review routes (non-admin). wf_delegations.view/edit = admin delegation management screen. After mig 156, self-service delegation INSERT/UPDATE requires wf_inbox.view (not just auth.uid()).
sec_permission_catalog vs sec_permission_matrix
sec_permission_catalog.view = read-only reference list of all permission codes. sec_permission_matrix.view/edit = the actual permission set builder. Before mig 149, permission_matrix writes were incorrectly gated on permission_catalog.edit — now cleanly separated.
sec_role_assignments.view + audit_log scope
After mig 159, sec_role_assignments.view only grants access to audit_log rows where entity_type IN ('user_roles', 'permission_set_assignments', 'permission_set_items'). Before mig 159, this permission gave unintended full audit_log read access.

Finds prev step via action_log (handles skipped steps correctly)
UPDATE workflow_tasks  status = 'returned'
UPDATE workflow_instances  current_step = prev_step
wf_resolve_approver() → delegation re-applied
INSERT workflow_tasks  (new task for prev approver, fresh SLA)
INSERT workflow_action_log  action = 'returned_to_previous_step'
📧 wf.returned_to_previous_step → Previous approver
✗ REJECT — wf_reject(task_id, reason)
Guards: task pending · instance active · you're assigned (or wf_manage.edit)
reason must not be blank
UPDATE workflow_tasks (this)  status = 'rejected'
UPDATE workflow_tasks (all other pending)  status = 'cancelled'
UPDATE workflow_instances  status = 'rejected'
wf_sync_module_status() → source record status = 'rejected'
📧 wf.rejected → Submitter (includes reason + step)
⇄ REASSIGN — wf_reassign(task_id, new_profile_id, reason)
Guards: task pending · you're assigned OR admin/wf_manage.edit
If step.allow_delegation: follow chain on new_profile_id (up to 5 hops)
UPDATE workflow_tasks (old)  status = 'reassigned'
INSERT workflow_tasks (new)  assigned_to = final_assignee

   metadata: from/to/final_assignee/delegation_applied
📧 wf.task_assigned → New assignee (reassigned=true)
📧 wf.task_removed → Old assignee (only if admin reassigned)

③ wf_advance_instance() — called after every approval
Find next active step where step_order > current AND not skip-evaluated
  │
  ├── MORE STEPS REMAIN
  │    ├── CC step? (is_cc = true)
  │    │    → INSERT task · 📧 wf.task_assigned → CC recipient
  │    │    → Auto-mark approved · INSERT action_log ('cc_notified')
  │    │    → RECURSE → wf_advance_instance() again
  │    └── Regular step
  │        → wf_resolve_approver() (delegation re-applied fresh)
  │        │
  │        ├── TIER 1 — Initiator-as-approver removal (always active)
  │        │    resolved = submitted_by?
  │        │    → NO task · INSERT action_log ('step_removed', task_id=NULL) · RECURSE
  │        │
  │        ├── TIER 2A — remove_duplicate_approver flag (template opt-in)
  │        │    next step resolves to same approver?
  │        │    → NO task · INSERT action_log ('step_removed', task_id=NULL) · RECURSE
  │        │
  │        ├── TIER 2B — skip_duplicate_approver flag (template opt-in)
  │        │    next step resolves to same approver?
  │        │    → INSERT task status='skipped' acted_at=now() · INSERT action_log ('skipped') · RECURSE
  │        │
  │        └── Normal — no removal/skip triggered
  │            → INSERT workflow_tasks (due_at = now() + sla_hours)
  │            → UPDATE workflow_instances current_step = next
  │            → 📧 wf.task_assigned → Next approver
  │            → STOP — wait for next approval
  │
  └── LAST STEP — all done
      → UPDATE workflow_instances  status = 'approved'
      → INSERT workflow_action_log  action = 'completed'
      → 📧 wf.completed → Submitter
      → wf_sync_module_status()
          expense_reports → UPDATE expense_reports  status/approved_at/approved_by
          profile_*       → UPDATE workflow_pending_changes  status/resolved_at
                          └── fires trg_apply_profile_pending_change
                                → upserts satellite table ✓
④ SLA Escalation — automatic, every 30 minutes (pg_cron)
wf_escalate_overdue_tasks() runs every 30 min via pg_cron
  Finds: status = 'pending' AND due_at < now() AND escalated_at IS NULL
  (SKIP LOCKED — safe for concurrent runs)
  │
  For each overdue task:
  ├── 📧 wf.sla_reminder         → Approver  "You have an overdue task"
  ├── Resolve approver's manager via employees.manager_id
  ├── 📧 wf.sla_escalation        → Approver's manager  "X has not actioned Y"
  │    (silently skipped if no manager found)
  └── UPDATE workflow_tasks  escalated_at = now()  ← one-shot flag, never fires again for this task
