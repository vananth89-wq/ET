Architectural Decisions

Why user_can() everywhere

Both has_role('admin') and has_permission(code) read from either user_roles or role_permissions. Since migration 107, role_permissions is no longer written to — has_permission() silently returns false for everyone. user_can() reads from permission_set_assignments → permission_set_items, which is what the Permission Matrix UI writes to.

No circular dependency

user_can() is SECURITY DEFINER — it runs as superuser and bypasses RLS entirely when reading permission_set_assignments, user_roles, target_groups, etc. Gating those tables on user_can() is safe.

Bootstrap guard on permission catalog

The five permission catalog tables use is_super_admin() OR user_can(...) — not user_can() alone. If an admin accidentally revokes their own catalog edit permission, a super admin (UUID allowlist, migration 112) can always recover. Regular user_can() alone would create a lockout scenario.

audit_log immutability

RLS UPDATE/DELETE policies were dropped and replaced with BEFORE triggers that RAISE EXCEPTION. Triggers cannot be bypassed by SECURITY DEFINER functions — only a superuser dropping the trigger can override. RLS alone is insufficient for true immutability.
